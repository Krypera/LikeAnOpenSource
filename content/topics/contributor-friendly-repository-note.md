# Write a repository note for a contributor-friendly project

This brief asks for one repository note about a real public project.

## What the final note should answer

- Does the README make the purpose of the project clear?
- Is setup realistic for a newcomer?
- Are contribution docs visible and usable?
- Is there evidence of issue hygiene or maintainer follow-through?
- What would a first useful contribution probably look like?

## Required evidence

The note should be grounded in things a reader can actually inspect:

- repository docs
- issue labels
- contribution guides
- release habits
- signs of maintainer feedback

## What to avoid

- generic praise like "great community"
- claiming something is beginner-friendly without showing why
- drifting into a feature review that ignores contribution reality
